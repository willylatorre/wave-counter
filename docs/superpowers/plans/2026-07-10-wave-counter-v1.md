# Wave Counter v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the self-hosted Wave Counter v1 library described in `docs/tickets/2026-07-10-wave-counter-v1-design.md`, including the canonical Rust/SQLite engine, Python and Node adapters, framework-neutral browser client, Vue component, playground, conformance tests, and release automation.

**Architecture:** A single Rust crate owns validation, migrations, SQLite transactions, totals, analytics, and structured errors. Thin PyO3 and napi-rs bindings expose that engine to FastAPI and Express; a framework-neutral TypeScript client consumes their shared HTTP contract, and the Vue package supplies the accessible visual component. Shared JSON fixtures and browser tests prove parity across the complete paths.

**Tech Stack:** Rust stable, rusqlite, chrono, uuid, PyO3/maturin, napi-rs, Python 3.10+, FastAPI, Node 20+, Express 5, TypeScript, Vue 3, Vitest, Playwright, npm workspaces, GitHub Actions.

## Global Constraints

- Counter keys match `[a-z0-9][a-z0-9_-]{0,63}`.
- Event IDs are UUIDv7 values generated by clients; duplicate IDs never increment twice.
- SQLite tables and migrations use the `waves_` namespace and request WAL mode, foreign keys, and a configurable busy timeout.
- Public timestamps are UTC ISO 8601 strings; stored timestamps are UTC Unix integers.
- V1 analytics accept only `window=7d` and return seven daily UTC buckets plus the previous seven-day comparison.
- Authentication, CORS, rate limiting, reset, deletion, listing, identity, arbitrary metadata, offline queues, React, Svelte, and the website are out of scope.
- Motion uses only opacity and transform, never blocks interaction, and is removed by `prefers-reduced-motion`.
- All public packages use one coordinated semantic version.
